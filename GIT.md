# PassMate Git 규칙

## 목적

이 문서는 PassMate 프로젝트에서 AI 에이전트와 함께 작업할 때 Git 사용을 안전하게 유지하기 위한 규칙이다.

## 기본 원칙

- 사용자가 요청하지 않으면 커밋하지 않는다.
- 사용자가 요청하지 않으면 push하지 않는다.
- 사용자의 변경사항을 임의로 되돌리지 않는다.
- 관련 없는 파일을 수정하거나 포맷팅하지 않는다.
- 작업 전후 변경 파일을 확인한다.

## 커밋

- 사용자가 명시적으로 요청할 때만 커밋한다.
- 커밋 전 변경 내용을 요약한다.
- 커밋 메시지는 한국어로 작성한다.
- 커밋 메시지는 변경 의도와 주요 변경 범위가 드러나도록 충분히 자세히 작성한다.
- 제목은 50자 안팎으로 간결하게 쓰고, 필요한 경우 본문에 핵심 변경 사항과 검증 내용을 적는다.
- 여러 성격의 변경이 섞이면 가능한 한 커밋을 분리한다.

예시:

```text
학습 세트 생성 흐름을 폴더 중심으로 개편

- 폴더 상세에서 학습 세트를 만들고 조회하는 화면을 추가
- 학습 세트와 카드를 한 트랜잭션으로 저장하는 API 추가
- npm run build, mvn clean test 통과
```

## 금지 명령어

아래 명령어는 사용자가 명시적으로 요청하고 위험을 이해한 경우가 아니면 실행하지 않는다.

```text
git reset --hard
git clean
git checkout .
git push --force
git push --force-with-lease
```

## 브랜치

- 개인 MVP 개발 단계에서는 main에서 직접 작업할 수 있다.
- 대규모 기능, 구조 변경, 실험 작업은 feature 브랜치를 권장한다.
- 브랜치 이름은 작업 의도를 드러내게 작성한다.

예시:

```text
feature/review-flow
feature/material-card-generation
fix/deck-delete-error
```

## 충돌과 변경사항

- 충돌이 발생하면 임의로 해결하지 말고 충돌 영역을 확인한다.
- 사용자가 만든 것으로 보이는 변경은 보존한다.
- 같은 파일에서 작업할 때는 기존 변경의 의도를 먼저 이해한다.

## 작업 완료 보고

작업 완료 시 다음을 요약한다.

- 수정한 파일
- 핵심 변경 내용
- 실행한 검증 명령
- 실행하지 못한 검증과 이유
- 남은 리스크
