# PassMate Frontend 규칙

## 목적

이 문서는 PassMate 프론트엔드의 Next.js/PWA 구현 규칙이다.

프론트엔드는 학습 흐름이 빠르고 명확해야 하며, 모바일에서도 자연스럽게 사용할 수 있어야 한다.

## 기본 원칙

- 실제 사용 가능한 화면을 먼저 만든다.
- 페이지는 얇게 유지하고 복잡한 UI는 컴포넌트로 분리한다.
- 비즈니스 로직과 API 호출은 hooks 또는 lib 계층으로 분리한다.
- 로딩, 빈 상태, 에러 상태, 성공 상태를 반드시 고려한다.
- 접근성 좋은 기본 HTML과 명확한 사용자 흐름을 우선한다.

## 페이지 구조

기본 라우트 구조는 다음을 따른다.

```text
app/
  deck/
  category/
  review/
  material/
  card/
  mypage/
```

라우트 이름은 사용자의 작업 흐름과 도메인을 기준으로 짓는다.

## 추천 폴더 구조

프로젝트 구조가 아직 확정되지 않았다면 다음 구조를 기본값으로 사용한다.

```text
src/
  app/
  components/
  features/
  hooks/
  lib/
  styles/
  types/
```

- `app`: 라우팅과 페이지 엔트리
- `components`: 공용 UI 컴포넌트
- `features`: 도메인 또는 기능별 UI와 로직
- `hooks`: 재사용 가능한 클라이언트 로직
- `lib`: API client, 유틸, 설정
- `types`: 공용 타입

## 컴포넌트

- 페이지 컴포넌트는 얇게 유지한다.
- 복잡한 UI는 `components` 또는 `features` 하위 컴포넌트로 분리한다.
- 컴포넌트 이름은 역할이 드러나게 짓는다.
- props는 구현 세부사항보다 의도를 표현하게 설계한다.
- 하나의 컴포넌트에 데이터 요청, 변환, 렌더링이 과하게 섞이지 않게 한다.
- 재사용되지 않는 컴포넌트는 해당 feature 가까이에 둔다.

## 상태 관리

- 가능하면 local state를 사용한다.
- 여러 컴포넌트가 공유해야 할 때만 상태를 위로 올린다.
- 전역 상태를 남용하지 않는다.
- 필요할 때만 Zustand를 사용한다.
- 서버 상태와 클라이언트 UI 상태를 구분한다.
- 계산 가능한 파생 데이터는 불필요하게 state로 저장하지 않는다.

## API 호출

- 화면에서 `fetch`를 직접 호출하는 것을 최소화한다.
- API client를 사용한다.
- 화면 컴포넌트 안에 API URL 문자열을 직접 작성하지 않는다.
- 요청과 응답 타입을 명확히 정의한다.
- 실패 응답은 사용자에게 이해 가능한 메시지로 변환한다.

## Empty State와 상태 표현

모든 주요 화면은 아래 상태를 고려한다.

- 로딩
- 빈 상태
- 에러 상태
- 성공 상태

빈 상태는 단순히 “없음”만 보여주지 말고, 사용자가 다음에 무엇을 하면 되는지 알려준다.

## UX

- 학습 흐름은 빠르고 방해가 적어야 한다.
- 중요한 행동 버튼은 명확하게 보이게 한다.
- 카드 복습 화면은 키보드와 모바일 터치 흐름을 모두 고려한다.
- 폼에는 라벨, 검증 메시지, 제출 중 상태가 있어야 한다.
- 삭제, 초기화 등 되돌리기 어려운 행동은 확인 절차를 둔다.

## 스타일링

- 기존 디자인 토큰과 스타일 패턴을 따른다.
- 간격, 폰트 크기, 색상 사용을 일관되게 유지한다.
- 반응형 레이아웃을 기본으로 고려한다.
- 화면 요소가 모바일에서 겹치거나 잘리지 않게 한다.
- 장식보다 가독성과 사용성을 우선한다.

## PWA

- 오프라인 경험은 MVP1에서 필요한 범위만 구현한다.
- 캐싱 전략은 데이터 정합성을 해치지 않게 신중히 적용한다.
- 설치 가능성보다 실제 학습 흐름의 안정성을 우선한다.

## 테스트

- 핵심 사용자 흐름은 가능하면 테스트한다.
- 중요한 컴포넌트 동작은 컴포넌트 테스트를 고려한다.
- 데이터 변환, 복습 로직 등 순수 로직은 단위 테스트를 작성한다.
- UI 변경은 가능한 경우 브라우저에서 직접 확인한다.
