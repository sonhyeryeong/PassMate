# PassMate Frontend 규칙

## 목적

이 문서는 PassMate 프론트엔드의 Next.js/PWA 구현 규칙이다.

프론트엔드는 학습 흐름이 빠르고 명확해야 하며, 모바일에서도 자연스럽게 사용할 수 있어야 한다.

## 기본 원칙

- 실제 사용 가능한 화면을 먼저 만든다.
- 페이지는 얇게 유지하고 복잡한 UI는 컴포넌트로 분리한다.
- 비즈니스 로직과 API 호출은 hooks 또는 lib 계층으로 분리한다.
- 로딩, 빈 상태, 에러 상태, 성공 상태를 반드시 고려한다.
- 접근성 좋은 기본 HTML과 명확한 사용자 흐름을 우선한다.
- PassMate는 모바일 앱 화면이 아니라 PWA 기반 웹 서비스 화면을 우선한다.
- 자료 입력과 관리는 데스크톱 웹에서 주로 사용한다고 가정한다.
- 모바일 PWA는 오늘의 복습, 폴더 조회, 내 정보 중심으로 축소한다.
- 사용자 화면에서는 Deck보다 폴더, Material보다 학습 세트, FlashCard보다 카드를 우선 사용한다.
- 카드 생성 UX는 퀴즐렛처럼 앞면/뒷면 카드 행을 빠르게 여러 개 추가하는 흐름을 우선한다.
- 사용자에게 보이는 모든 화면 문구는 한글을 기본으로 작성한다.
- UI 문구는 내부 Entity, API, 개발자 용어가 아니라 사용자의 역할과 행동 기준으로 작성한다.
- 예: `Create Material` 대신 `학습 세트 만들기`, `FlashCard` 대신 `카드`, `ReviewHistory` 대신 `복습 기록`을 사용한다.

## 페이지 구조

기본 라우트 구조는 다음을 따른다.

```text
app/
  page.tsx
  folder/
  review/
  mypage/
```

라우트 이름은 사용자의 작업 흐름과 도메인을 기준으로 짓는다.

`page.tsx`는 Dashboard로 사용한다.

학습 세트와 카드는 독립 최상위 메뉴나 독립 최상위 라우트로 만들지 않는다. 아래처럼 폴더 상세 흐름 안에서 접근한다.

```text
app/
  folder/
    page.tsx
    [folderId]/
      page.tsx
      set/
        new/
          page.tsx
        [setId]/
          page.tsx
          edit/
            page.tsx
```

## 추천 폴더 구조

프로젝트 구조가 아직 확정되지 않았다면 다음 구조를 기본값으로 사용한다.

```text
src/
  app/
  components/
    layout/
    folder/
    study-set/
    flashcard/
    review/
    common/
  features/
  hooks/
  lib/
  styles/
  types/
```

- `app`: 라우팅과 페이지 엔트리
- `components`: 공용 UI 컴포넌트
- `components/layout`: AppShell, Header, Sidebar, MobileBottomNav, PageHeader, Breadcrumb
- `features`: 도메인 또는 기능별 UI와 로직
- `hooks`: 재사용 가능한 클라이언트 로직
- `lib`: API client, 유틸, 설정
- `types`: 공용 타입

## 화면 레이아웃

- 기준 화면은 1440px 데스크톱 웹이다.
- 데스크톱 기본 구조는 Header, Sidebar, Main Content이다.
- 주요 메뉴는 대시보드, 폴더, 복습, 내 정보이다.
- Notion, Linear, GitHub Projects처럼 관리형 웹 서비스 느낌을 우선한다.
- 데스크톱에서는 넓은 작업 공간을 활용하고, 한 줄 모바일 카드 나열처럼 보이지 않게 한다.
- Sidebar는 데스크톱에서 표시하고 모바일에서는 숨긴다.
- 모바일에서는 Bottom Navigation을 사용할 수 있다.
- Review 화면은 웹과 모바일 모두에서 카드 중심으로 단순하게 유지한다.
- 학습 세트 생성/수정은 카드 앞면/뒷면을 여러 행으로 입력하는 테이블형 편집 UI를 우선한다.
- 카드 단일 생성/수정은 데스크톱에서 모달 또는 우측 패널을 우선한다.
- 카드를 학습 세트 없이 독립 생성하는 흐름은 만들지 않는다.

## 컴포넌트

- 페이지 컴포넌트는 얇게 유지한다.
- 복잡한 UI는 `components` 또는 `features` 하위 컴포넌트로 분리한다.
- 컴포넌트 이름은 역할이 드러나게 짓는다.
- props는 구현 세부사항보다 의도를 표현하게 설계한다.
- 하나의 컴포넌트에 데이터 요청, 변환, 렌더링이 과하게 섞이지 않게 한다.
- 재사용되지 않는 컴포넌트는 해당 feature 가까이에 둔다.

## 상태 관리

- 가능하면 local state를 사용한다.
- 여러 컴포넌트가 공유해야 할 때만 상태를 위로 올린다.
- 전역 상태를 남용하지 않는다.
- 필요할 때만 Zustand를 사용한다.
- 서버 상태와 클라이언트 UI 상태를 구분한다.
- 계산 가능한 파생 데이터는 불필요하게 state로 저장하지 않는다.

## API 호출

- 화면에서 `fetch`를 직접 호출하는 것을 최소화한다.
- API client를 사용한다.
- 화면 컴포넌트 안에 API URL 문자열을 직접 작성하지 않는다.
- 요청과 응답 타입을 명확히 정의한다.
- 실패 응답은 사용자에게 이해 가능한 메시지로 변환한다.

## Empty State와 상태 표현

모든 주요 화면은 아래 상태를 고려한다.

- 로딩
- 빈 상태
- 에러 상태
- 성공 상태

빈 상태는 단순히 “없음”만 보여주지 말고, 사용자가 다음에 무엇을 하면 되는지 알려준다.

## UX

- 학습 흐름은 빠르고 방해가 적어야 한다.
- 사용자가 지금 무엇을 만들고, 어디로 이동하고, 다음에 무엇을 해야 하는지 한글 문구만으로 이해할 수 있어야 한다.
- 중요한 행동 버튼은 명확하게 보이게 한다.
- 카드 복습 화면은 키보드와 모바일 터치 흐름을 모두 고려한다.
- 폼에는 라벨, 검증 메시지, 제출 중 상태가 있어야 한다.
- 삭제, 초기화 등 되돌리기 어려운 행동은 확인 절차를 둔다.

## 스타일링

- 기존 디자인 토큰과 스타일 패턴을 따른다.
- 간격, 폰트 크기, 색상 사용을 일관되게 유지한다.
- 반응형 레이아웃을 기본으로 고려한다.
- 화면 요소가 모바일에서 겹치거나 잘리지 않게 한다.
- 장식보다 가독성과 사용성을 우선한다.

## PWA

- 오프라인 경험은 MVP1에서 필요한 범위만 구현한다.
- 캐싱 전략은 데이터 정합성을 해치지 않게 신중히 적용한다.
- 설치 가능성보다 실제 학습 흐름의 안정성을 우선한다.

## 테스트

- 핵심 사용자 흐름은 가능하면 테스트한다.
- 중요한 컴포넌트 동작은 컴포넌트 테스트를 고려한다.
- 데이터 변환, 복습 로직 등 순수 로직은 단위 테스트를 작성한다.
- UI 변경은 가능한 경우 브라우저에서 직접 확인한다.
